ts-react-quality-lens — Improvement Plan

A gap analysis benchmarking ts-react-quality-lens against its sibling rust-quality-lens, plus TypeScript/React-specific opportunities the Rust lens does not need. The goal: bring the two lenses to contract and rigor parity so a single dashboard can consume both, then push the TS lens further where its richer toolchain allows.

Compared against rust-quality-lens @ master (README, docs/risk-model-v1.md, docs/measurement-upgrades.md, docs/extraction-notes.md, src/*.rs). TS side reviewed across all of src/ (27 modules). Date: 2026-06-03.

Headline: the TS lens already matches or exceeds the Rust lens on extraction (real TS compiler API, framework adapters, hooks linting). It lags significantly on the parts that make a lens trustworthy and portable: a versioned, externalized risk model, a real architecture risk map with category scores, missing-vs-stale confidence discipline, and git-history change risk. Several catalog descriptions also over-promise capabilities the code does not yet implement.

Gap analysis

G1 Externalize and version the risk model High

rust-quality-lens

Weights, caps, and classification thresholds live in src/risk_model.rs, documented in docs/risk-model-v1.md under model id rqlens.architecture_risk v1. Raw facts are kept separate from derived scores. Every scored record carries risk_model_id, risk_model_version, risk_calibration, and the map emits meta.risk_model_weights + meta.risk_model_tool_scores.

ts-react-quality-lens

Weights are hardcoded magic numbers scattered across scoring.ts (fn.complexity * 8 + nesting * 5 …), graph.ts, dependency.ts, and quality.ts. Thresholds (70/35) live in riskForScore. No version, no document, no calibration metadata in artifacts, no raw/derived separation.

Plan: create src/risk-model.ts exporting a frozen RISK_MODEL object (id: "tsrqlens.architecture_risk", version: 1) holding all weights/caps/thresholds and per-tool calibration tables. Replace inline constants in scoring.ts/graph.ts/quality.ts/dependency.ts with lookups. Attach risk_model_id/risk_model_version/risk_calibration to scored records and meta.risk_model_* to map.json. Document it in docs/risk-model-v1.md mirroring the Rust doc. This is the foundation the rest of the gaps build on.

G2 Turn the architecture map into a real risk model High

rust-quality-lens

Per-module derived categories: maintainability_risk, change_risk, performance_risk, correctness_risk, architectural_risk, quality_risk, and a total_score that is null if any required category is unknown. Architectural risk folds in cycle-membership penalty, layer violations, and a large-module penalty. Classification thresholds map score → warning/bad.

ts-react-quality-lens

mapNode just copies maxScoreFor(...) out of other artifacts into category_scores. correctness is a binary 0/40 hack; there is no change from git, no cycle/layer penalty feeding the node, no total_score, and missing inputs silently collapse to 0 (because maxScoreFor returns 0), inflating apparent health.

Plan: add architectureRiskScores(module, facts) in the new risk-model module producing the six categories + total_score, with null (not 0) for unavailable inputs and an unknown_metrics array explaining why. Fold cycle membership, layer violations (G6), and a large-module penalty into architectural_risk; fold git change facts (G4) into change_risk; derive correctness_risk from real test status rather than 0/40. Add a classification (ok/warning/bad) and surface unknown-module counts in map.json.summary.

G3 Distinguish missing from stale, and deepen confidence High

rust-quality-lens

Confidence carries complete, partial, confidence_scope, required_inputs, observed_inputs, missing_input, stale_input, unsupported_pattern. The map never substitutes 0 for a required missing/stale input — it marks the field unknown and records it in meta.summary.artifact_status.

ts-react-quality-lens

createConfidence is a flat boolean bag (tsconfig_found, git_history_available, …). The map's artifact_status only reports "available"/"missing" — no staleness detection at all: a map.architecture run will happily consume a hotspots.json written before the source changed.

Plan: extend the Confidence type with required_inputs/observed_inputs/confidence_scope and a complete|partial flag. Stamp each artifact with a content hash + measured_at (already present) and the source-set hash it was computed from; in measureArchitectureMap, compare each input artifact's source hash against the current project to mark stale vs available vs missing, and propagate null for derived fields (depends on G2).

G4 Git history: defect-keyword + co-change → change risk Medium

rust-quality-lens

change_risk is built from churn, commit count, contributor count, defect-keyword commits (fix/bug/revert signals), and co-change (files that change together), with a missing-test-evidence penalty.

ts-react-quality-lens

history.ts computes only churn (commits + contributors) and it is used only as a small capped term inside locality. There is no defect-keyword mining, no co-change graph, and no change_risk category anywhere.

Plan: extend gitChurn into a gitHistory producer that also returns defect-keyword commit counts (regex over commit subjects: fix|bug|revert|hotfix|regression) and a co-change map (files appearing in the same commit). Feed these into change_risk in the map (G2) using calibrated weights from the risk model (G1). Keep raw counts in the artifact separate from the derived score.

G5 Add an AST/structural clone layer Medium

rust-quality-lens

Two clone engines: token (normalized windows) and ast (a helper hashes functions/methods, normalizing local names and literals while keeping operator/call/type structure, skipping trivial functions below a node threshold, with a cross-file factor).

ts-react-quality-lens

Only token-level: jscpd plus a normalized line-window fallback. No structural layer — even though the TS compiler AST is already loaded for every module, so the data needed for structural hashing is sitting unused.

Plan: add an engine: "ast" layer to clones.json. Walk each module's astSourceFile, hash function/method bodies after normalizing identifiers and literals (reuse stableHash from clone-utils), drop bodies under a min-node threshold, and emit groups where ≥2 non-trivial functions share a hash. Tag token rows confidence_scope: "source_scan" and AST rows "syntax_facts", matching the Rust split.

G6 Layer classification & layer-violation detection Medium

rust-quality-lens

A generic layer classifier assigns each module a layer; cross-layer edges that violate the intended direction become layer violations that feed architectural_risk.

ts-react-quality-lens

No layer concept exists. dependency.ts detects cycles, deep relative imports, and barrels, but the catalog text for dependency_health and leverage already claims "layer violations" and "package boundaries" (see G7).

Plan: add a lightweight, config-driven layer classifier (e.g. ordered globs: app/route → feature → ui → hooks → lib/util, plus a generic fallback by directory depth/role). Flag edges that import "upward" (util → feature, ui → route) as layer_violation records in dependency_health.json and feed them into architectural_risk. Make the layer map an optional config field so it stays portable.

G7 Reconcile catalog promises with implementation Medium

The Rust lens keeps its catalog descriptions tightly aligned to what producers actually emit. The TS catalog (src/tasks.ts) advertises capabilities the code does not implement:

  • locality_dynamic claims "contributors, and ownership spread" — only churn commits/contributors are computed; no ownership-spread metric.
  • locality_leverage claims "co-change ripple" — no co-change is computed (G4).
  • dependency_health claims "layer violations, package boundaries" — neither exists yet (G6).

Plan: for each claim, either implement it (folded into G4/G6) or trim the description so consumers and the dashboard are not misled. A catalog that over-promises silently degrades trust in every other number the lens reports. Add a test asserting each catalog capability keyword maps to a field actually present in that task's artifact.

G8 Surface unsupported patterns instead of dropping them Medium

rust-quality-lens

Macro-generated modules/tests and other unparseable constructs are reported as explicit unsupported_pattern confidence signals — "unsupported is not the same as absent."

ts-react-quality-lens

The import/extract fallback silently skips what it cannot resolve: tsconfig paths aliases, re-export chains (export * from), dynamic/ non-literal specifiers, and barrel indirection. These vanish from the dependency graph with no signal, quietly understating coupling and cycles.

Plan: when the built-in parser hits a path alias, computed specifier, or re-export it cannot resolve, push an unsupported_pattern entry into confidence (kind + file + line) rather than dropping it. Resolve tsconfig paths where the compiler program is loaded so aliased imports become real edges; otherwise mark them unresolved-but-known.

G9 Assertion-backed golden fixtures Medium

rust-quality-lens

A golden fixture crate deliberately exercises edge cases (grouped imports, duplicate test names, same-named types in different modules, pub use, macro tests, suppression de-dup, attribute module wiring) so the upgraded behavior is executable and regression-proof.

ts-react-quality-lens

examples/basic is a demonstration app, not a golden fixture with pinned expected outputs. test/cli.test.ts exercises the CLI but the corpus doesn't cover the hard cases the TS lens will meet in real repos.

Plan: build a test/fixtures/golden project covering: path aliases, barrel re-exports, export * from, dynamic imports, import cycles, duplicate component names across files, colocated vs external tests, "use client"/server-only boundary, and a deliberate structural clone pair. Snapshot the produced artifacts (with volatile fields like measured_at/ host normalized) and assert on them, so every gap fixed above is locked in.

G10 Optional performance input contract Low

rust-quality-lens

performance_risk is derived from an optional slowspots.json (bench mean/ variance). Absent ⇒ category stays unknown, rest of the map remains usable.

ts-react-quality-lens

No runtime/bundle performance input. react_health has a "render_performance" heuristic but nothing measures actual cost.

Plan: define an optional input artifact (e.g. bundle_stats.json / render_costs.json) the lens reads if present — per-module bundle bytes from a stats file, or render timings — feeding a performance_risk category that is null when absent. Keep it strictly optional and project-supplied, exactly like the Rust slowspots model.

2. TypeScript/React-only opportunities

Areas with no Rust analogue where the TS lens can lead, leveraging the compiler API and JS ecosystem it already depends on.

  • tsconfig strictness scoring. compilerOptionSummary already captures strict, noUncheckedIndexedAccess, exactOptionalPropertyTypes, etc., but nothing scores them. Emit a project-level "type-safety posture" score so a non-strict project is visibly penalized. High value, low cost
  • Dead/unused exports. The compiler program + import graph already know every export and every consumer — flag exported symbols with zero internal importers (likely dead code or untested public surface).
  • Accessibility heuristics. Optionally integrate eslint-plugin-jsx-a11y the same way eslint-plugin-react-hooks is wired, adding an a11y dimension to react_health.
  • Prop-drilling / context overuse depth. Extend hiddenCouplingSignals (already detecting createContext/useContext) into a measured prop-drilling depth signal.
  • Import cost / barrel fan-out impact. Barrels are detected; quantify the transitive fan-out they introduce to estimate bundle/tree-shaking risk.
  • Server/client boundary depth. The boundary check exists; extend it to flag client components transitively importing server-only modules, not just direct co-occurrence.

3. Internal code-quality cleanups

  • Centralize magic numbers (subsumed by G1) — every * 8, * 12, Math.min(100, …), and 70/35 threshold should resolve through the risk model so calibration is one file, not nine.
  • Confidence as a typed shape, not Record<string, boolean> — gives consumers a stable contract and makes G3 enforceable by the type checker.
  • Separate raw-fact extraction from scoring in the producers, mirroring the Rust facts.rs / risk_model.rs split, so future model versions rescore without re-extracting.
  • Schema/version discipline: bump SCHEMA_VERSION deliberately when artifact shapes change (the above changes will), and document the contract the dashboard relies on.

4. Strengths to preserve

  • Real TypeScript compiler API integration — type health, diagnostics, and inferred export types go beyond what regex/syn extraction gives the Rust lens.
  • Framework adapters (Next app/pages, Remix, Vite, Storybook, client/server directives) — genuinely portable React awareness.
  • Graceful tool degradation — jscpd, dependency-cruiser, and eslint-react-hooks all have explainable fallbacks with tool_status recorded. Keep this as the new confidence work lands.
  • Cached AnalysisContext — the project is parsed once and shared, analogous to the Rust RunContext. Extend it to also cache git history and the new facts.

5. Phased roadmap

Ordered so each phase unblocks the next. Phase 1 is the trust foundation; nothing downstream is reliable without it.

Phase 1 — Trust foundation do first

  1. G1 Externalize the risk model into src/risk-model.ts + docs/risk-model-v1.md; attach model id/version/calibration to records and map meta.
  2. G3 Typed confidence with required/observed inputs + complete/partial; artifact source-hash stamping.
  3. G2 Real category scores + total_score with null/unknown_metrics discipline (replaces the 0-substitution and 0/40 correctness hack).
  4. G3 Staleness detection in map.architecture (missing vs stale vs available).

Phase 2 — Close the measurement gaps next

  1. G4 Git history: defect-keyword + co-change → change_risk.
  2. G6 Layer classifier + layer violations → architectural_risk.
  3. G5 AST structural clone layer.
  4. G8 Unsupported-pattern signals + tsconfig paths resolution.
  5. G7 Reconcile catalog descriptions with implemented fields (+ guard test).

Phase 3 — Hardening & TS-native leadership then

  1. G9 Golden fixture project + normalized artifact snapshots.
  2. tsconfig strictness posture score & dead-export detection (Section 2).
  3. Optional a11y integration; prop-drilling depth; import-cost/fan-out.
  4. G10 Optional performance input contract.
