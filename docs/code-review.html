ts-react-quality-lens — Opportunities to Improve

A code review of the measurement engine: what is solid, what is risky, and where the highest-leverage improvements are. This document does not change any code — it is an inventory of opportunities.

Scope: all of src/ (24 modules), bin/, scripts/, test/, build config, and CI. Reviewed at commit 672b921 (v0.1.0). Date: 2026-06-02.

1. Executive summary

ts-react-quality-lens is a well-structured analysis engine. The artifact contract, the confidence metadata, the graceful tool fallbacks, and the atomic artifact writes are genuinely good design and should be preserved. The biggest single opportunity is also the most ironic one: a library whose job is measuring TypeScript quality ships with type checking effectively turned off and almost no type annotations of its own. Beyond that, there are a handful of correctness bugs in the heuristics, a few performance cliffs on large repos, and a build/CI pipeline that does redundant work.

What to do first: turn on strict and type the core data shapes (#1). It is the change that pays back everywhere else — it will surface several of the other bugs in this list for free, and it makes the emitted .d.ts a real contract for consumers instead of a wall of any.

2. What is already good

  • Confidence as a first-class output. createConfidence / analysisConfidence (src/config.ts:91, src/analysis-context.ts:40) make "we couldn't measure this" visible to consumers instead of silently producing zeros. This is the right instinct for an explainable lens.
  • Graceful tool degradation. runToolAdapter (src/integrations.ts:239) gives every external tool a uniform { available, ran, reason } envelope and a built-in fallback. jscpd, dependency-cruiser, and eslint-plugin-react-hooks are all optional.
  • Atomic, deterministic artifacts. writeArtifact (src/writer.ts:4) writes to a temp file then renames; records are sorted by score then id so diffs are stable.
  • Clean separation. Pure scoring (src/scoring.ts), extraction (src/extract.ts), and per-task measures (src/measures/*) are nicely isolated, and the analysis context memoizes the expensive project parse so measure all parses once.
  • Cross-platform care. Windows .cmd quoting and case-insensitive path matching (src/integrations.ts:387,436) show real attention to detail.

3. Type safety High

3.1 Type checking is turned off in a TypeScript quality tool

tsconfig.json:12 — "strict": false

The tool flags weak typing in the code it measures (quality.escape_hatches, quality.type_health), yet its own tsconfig sets strict: false. Combined with the fact that almost no function declares parameter types — e.g. runCli(argv) (src/cli.ts:13), runMeasure(config, taskId, command, ...) (src/cli.ts:49), loadConfig(configArg) (src/config.ts:46), analyzeModule(config, file, tsProject) (src/extract.ts:14) — the entire engine runs as implicit any. The compiler is currently catching almost nothing.

Recommend: enable strict (or at minimum noImplicitAny + strictNullChecks) and introduce named interfaces for the core shapes — Config, SourceFile, ModuleRecord, FunctionRecord, Artifact, Confidence. Do it incrementally: turn the flag on, fix the anys in config.ts/files.ts/extract.ts first, and let the types flow outward.

3.2 The emitted .d.ts contract is untyped

package.json:23 (ships dist/), tsconfig.json:10 (declaration: true)

The package is described as a "contract" producer, and it emits declaration files. But because everything is implicit any, the generated .d.ts exports functions with any parameters and any returns. A consumer importing analyzeProject or the artifact types gets no real contract — only the JSON shape, which lives only in the ts-react-quality-lens.schema.json file.

Recommend: once #3.1 lands, export the artifact and record types from a single src/types.ts and re-export them from the package entry so both the runtime engine and external consumers share one definition. This is what makes the "stable contract" claim true at the type level.

4. Correctness of the heuristics

4.1 Regex any/assertion counts overcount Medium

src/extract.ts:23-27 (escapeCounts), consumed in src/measures/quality.ts:100 (weakSurface)

escapeCounts.any is countMatches(file.text, /\bany\b/g) over raw source — it matches the word any inside comments, string literals, and identifiers (company does not match due to \b, but "any value" in a doc comment or a JSX text node does). The assertion regex similarly matches as in places that are not type assertions. These inflated counts feed leverage.weak_surface, which can wrongly demote a clean module to high risk. Note the AST-based escapeRecords (src/scoring.ts:47) already does this correctly via the compiler — the regex path is the weak one.

Recommend: derive weakSurface from the AST escape-hatch records you already compute, rather than from the regex escapeCounts. That removes the false positives and deletes a redundant scan.

4.2 Passing tests are recorded as "unknown" Medium

src/measures/correctness.ts:35

status: execution.status === "passed" ? "unknown" : "not_run"

After correctness.all runs the suite and it passes, every per-test entry in test_catalog.json is labelled "unknown" rather than "passed". The root cause is that runTestCommand (src/correctness.ts:25) only captures the aggregate process exit code, so it genuinely cannot attribute pass/fail to individual tests — the ternary is papering over a missing capability. The result reads as backwards to a consumer.

Recommend: either (a) be honest and report a single suite-level status, leaving per-test status as not_run; or (b) parse the runner's machine-readable output (Vitest --reporter=json, Jest --json, node --test --reporter=tap) and attach real per-test results. Option (b) is the high-value version and fits the "explainable evidence" goal.

4.3 gitChurn uses a fragile filename/author heuristic over full history Low

src/history.ts:7-25

git log --name-only --format=%an is parsed by treating any line that matches /\.[cm]?[jt]sx?$/ as a filename and every other non-empty line as an author. An author whose display name ends in .ts would be misread as a file; a path that does not end in a JS/TS extension is silently dropped. It also walks the entire repository history on every run and never restricts to source_roots, which is slow on large/old repos and counts churn for files that have since been deleted or renamed.

Recommend: use a record separator that cannot collide with paths (--format=%x00%an%x00 or -z), scope with -- <source_roots>, and consider a --since window so churn reflects recent activity rather than all of history.

4.4 Multi-line named exports defeat barrel detection Low

src/extract.ts:236-242

isBarrel requires every non-comment line to start with export . A barrel written as export {\n a,\n b\n} from "./x"; fails the check because the member lines do not start with export, so it is not classified as a barrel and escapes barrel_fanout scoring in dependency health.

Recommend: classify barrels from the AST (you already parse it) — a module whose statements are all ImportDeclaration/ExportDeclaration re-exports — instead of line-prefix matching.

4.5 package_manager_detected confidence is effectively always true Low

src/config.ts:43 (rule list includes package.json), src/config.ts:96

The npm rule matches on the presence of package.json, which exists in essentially every project. So detectPackageManager almost never returns "unknown", and the package_manager_detected confidence flag is true even when the real package manager could not be determined (e.g. a project with no lockfile at all). The signal is not telling consumers what it claims.

Recommend: treat a lockfile as the detection signal and package.json alone as "npm assumed, low confidence", so the flag distinguishes a real detection from a default.

5. Performance & scalability

5.1 Cycle detection is exponential High

src/graph.ts:51-75 (findCycles)

The DFS starts a fresh traversal from every node, and on each step it clones the visited set (new Set(seen)) and the path array, exploring all distinct paths rather than all edges. On a densely connected internal import graph this is worst-case exponential in the number of paths, not linear in edges. For a small example app it is invisible; on a real monorepo it can hang. (And when dependency-cruiser is available its cycles are preferred anyway, so this expensive fallback may run only to be deduped away.)

Recommend: replace with Tarjan's strongly-connected-components (linear time) to enumerate cycles, or guard the existing DFS with a path-length/visited-node budget. At minimum, skip the built-in cycle search entirely when dependency-cruiser ran successfully.

5.2 Redundant clone engines and re-derivations Low

src/measures/clones.ts:13-42, src/graph.ts:17-31

The built-in line-window clone scan always runs, even when jscpd succeeded; the results are then merged and de-duped (dedupeCloneRegions). That is double work on every file. Separately, mapNode recomputes correctnessMap(artifacts.correctness) once per node (src/graph.ts:27), rebuilding the same map N times.

Recommend: skip the heuristic clone pass when jscpd.ran is true (it is a fallback, not an ensemble), and hoist correctnessMap out of the per-node loop in measureArchitectureMap.

5.3 Whole-project in-memory model with no caching Info

src/analysis-context.ts:12-20

analyzeProject reads every source file's full text, splits it into a lines array, and builds a TS Program with full getPreEmitDiagnostics — all held in memory at once. This is fine for the example app and is correctly memoized within a single measure all run, but there is no cross-run incremental cache, so re-measuring a large repo re-parses everything from scratch each time.

Recommend (later): for the "monorepos from the start" goal in the plan, consider a content-hash cache keyed on file mtime/hash so unchanged files skip re-analysis between runs.

6. Build, tooling & DX Medium

6.1 npm run ci rebuilds the project four times

package.json:12-15

"ci":           "npm run lint && npm run build && npm run format:check && npm run test"
"lint":         "tsc -p tsconfig.json --noEmit"   // a build (type check)
"build":        "tsc -p tsconfig.json"
"format:check": "npm run build && node dist/scripts/format-check.js"
"test":         "npm run build && node --test dist/test/cli.test.js"

lint type-checks (a full compile), build compiles again, then format:check and test each run build a third and fourth time. CI does roughly 4× the compiler work it needs to.

Recommend: build once, then run the downstream steps against that output — e.g. "ci": "npm run lint && npm run build && node dist/scripts/format-check.js && node --test dist/test/cli.test.js", and drop the inner npm run build from format:check/test (or keep thin standalone variants for local use).

6.2 "lint" is a misnomer — there is no linter for this repo

package.json:13, unused scripts/lint.ts

The lint script is tsc --noEmit; it does no linting. There is a separate scripts/lint.ts that runs node --check on emitted JS, but it is wired into nothing. The package depends on eslint and @typescript-eslint/parser, yet ships no ESLint config for its own sources — so the tool that lints other people's React code does not lint itself.

Recommend: rename the type-check script to typecheck, and add a real lint backed by a flat ESLint config (you already have the dependencies). This also lets you dogfood eslint-plugin-react-hooks-style rules on the codebase.

6.3 Per-run temp directories are never cleaned Medium

src/integrations.ts:124 (jscpd), src/integrations.ts:164 (eslint config)

Both jscpd output and the generated eslint flat-config are written to output_dir/.tooling/<tool>-<pid>-<timestamp> and never removed. Every measure invocation leaves a new directory behind, so .tooling grows without bound.

Recommend: write these under the OS temp dir (fs.mkdtempSync(path.join(os.tmpdir(), ...))) and remove them in a finally block, or sweep output_dir/.tooling at the start of a run.

7. Robustness & contract

7.1 The config schema exists but is never enforced Medium

ts-react-quality-lens.schema.json (shipped), src/config.ts:46-89 (loadConfig)

A JSON Schema for the config is authored and published in the package's files list, but loadConfig just JSON.parses the file and reads fields with ?? defaults. A typo like "source_root" (singular) or a wrong type for exclude is silently ignored — the user gets empty/odd results with no error. The schema is documentation that the code does not honor.

Recommend: validate the parsed config against the schema (e.g. a tiny hand-rolled check or a dependency-free validator) and emit a clear error listing unknown/invalid keys. Failing loudly on a misconfigured run is friendlier than silently measuring nothing.

7.2 Argument parsing is permissive and silent Low

src/cli.ts:74-89 (parseArgs)

--config with no following value yields undefined (then falls back to the default path); unknown flags are pushed into positionals and may be mistaken for a task id. There is no "unknown task" hint that lists valid ids — the error is thrown from deep in runMeasure.

Recommend: validate that --config has a value, reject unknown flags, and on an unknown task id print the available ids (you already have TASKS).

7.3 Schema version is hardcoded equal to the package version Low

src/tasks.ts:2 (SCHEMA_VERSION = "0.1.0"), package.json:3

The artifact schema_version is a literal that happens to equal the package version. These will drift: a routine package bump that does not change the artifact shape will nonetheless advance the "schema version" consumers key off, and vice-versa.

Recommend: version the artifact schema independently of the npm package, and bump it only when the artifact contract actually changes.

8. Testing Low

8.1 One integration test, environment-dependent

test/cli.test.ts

The suite is essentially one end-to-end measure all test plus two targeted cases. It asserts that jscpd, dependency-cruiser, and eslint-react-hooks ran (test/cli.test.ts:66-74) — which couples the test to those binaries being installed and behaving identically across OSes. The pure, high-value logic — riskForScore thresholds, normalizeCloneLine, canonicalCycle rotation, the complexity counters in ast-metrics.ts, findCycles — has no direct unit coverage. CI also runs only on ubuntu-latest, so the Windows-specific cmd.exe quoting path is never exercised.

Recommend: add focused unit tests for the pure scoring/AST helpers (fast, deterministic, no external tools), and gate the "tool ran" assertions behind a capability check so the core suite passes without optional binaries. Consider adding a Windows job to the CI matrix given the platform-specific code.

9. Security Low

9.1 Config-supplied test command runs through a shell

src/correctness.ts:28 (execSync(config.testCommand, …))

correctness.all executes test_command from the config file via execSync (full shell interpretation). For the intended use — a developer measuring their own repo — this is fine. But if a config ever arrives from an untrusted source (CI artifact, third-party repo, a dashboard that ingests external configs), arbitrary command execution follows. The other tool runners correctly use execFileSync with arg arrays; this one is the outlier.

Recommend: document the trust boundary ("the config is trusted input"), and make correctness.all opt-in/explicit so test execution never happens implicitly on an unfamiliar config. If feasible, run the command via execFileSync with a parsed argv instead of a shell string.

10. Minor polish

  • type_information_available is set to false in createConfidence (src/config.ts:101) and then overwritten in analysisConfidence (src/analysis-context.ts:46) — harmless duplication, but the default is dead.
  • existingAbsoluteRoots (src/integrations.ts:405) exists only to be wrapped by existingRelativeRoots; could be inlined.
  • measure-support.ts is a large re-export barrel — the very pattern the dependency-health measure flags as barrel_fanout. Worth a knowing comment, or splitting, for dogfooding's sake.
  • analyze.ts (src/analyze.ts) is a two-line re-export; confirm it is a real public entry point and not a leftover, since package.json main points at cli.js.
  • Both examples/basic/src/format.test.js and format.test.ts exist — intentional fixture duplication, but worth a one-line note so it is not mistaken for an accident.
