TypeScript React Quality Lens Plan

This document is the build plan for ts-react-quality-lens, a reusable measurement engine for TypeScript and React projects. It mirrors the successful boundaries of rust-quality-lens while adapting the measurements to TypeScript, React, frontend architecture, and modern JavaScript tooling.

1. Purpose

The goal of ts-react-quality-lens is to produce structured, explainable JSON measurements for TypeScript and React codebases. It should identify quality risks, correctness gaps, architecture pressure, type-system weakness, React-specific hazards, dependency issues, and change hotspots.

Like rust-quality-lens, this repository should not own a dashboard, web server, or project UI. Its job is to act as a reusable producer of analysis artifacts. The dashboard and local task runner should remain in project-management-board.

2. Repository Boundary

The new lens should live as a sibling repository rather than inside rust-quality-lens. The two projects should share a contract, not an implementation.

  • rust-quality-lens: reusable Rust quality, correctness, locality, leverage, and architecture-map producers.
  • ts-react-quality-lens: reusable TypeScript, JavaScript, JSX, TSX, and React quality producers.
  • project-management-board: dashboard, task catalog consumer, local run API, and visualization host.

If common behavior emerges across lenses, it can later be extracted into a shared contract package. That should wait until duplication is clear and stable.

3. Design Principles

  • Produce stable machine-readable artifacts, not prose reports only.
  • Keep all measurements explainable through visible evidence and signals.
  • Prefer project-native signals such as tsconfig, package scripts, test runner config, and framework conventions.
  • Do not require a specific React framework.
  • Support monorepos from the start where reasonable.
  • Distinguish missing evidence from healthy evidence.
  • Preserve unknown, partial, stale, or unsupported states in artifact metadata.
  • Avoid hardcoding dashboard task metadata in measured projects.
  • Let the dashboard consume the lens catalog exactly as it does for Rust.

4. Core Contract

The new lens should provide the same high-level interface as rust-quality-lens.

  • A project-local config file.
  • A catalog command that prints board-compatible task metadata.
  • A measure command that runs one task or all tasks.
  • Stable task IDs.
  • Stable artifact names.
  • Output artifacts under a configured analysis directory.
  • Measurement confidence metadata in every major artifact.
  • Source provenance: command, host, measured timestamp, and source type.

5. Configuration Model

The first version should support explicit configuration while also providing useful auto-detection.

Concept Purpose
project_name Human-readable project name used in catalog and map metadata.
project_root Measured project root. Relative paths resolve from the config file.
source_roots Directories to analyze, commonly src, app, pages, components, packages, or libs.
test_roots Directories and colocated test patterns used by the correctness catalog.
output_dir Artifact output directory. Prefer target/analysis for consistency with rust-quality-lens.
tsconfig Main TypeScript project config. Auto-detect tsconfig.json when omitted.
package_manager npm, pnpm, yarn, bun, or auto-detected from lockfiles.
framework React framework, such as Vite, Next.js, Remix, Expo, Astro React, or auto.
test_runner Vitest, Jest, Playwright, Cypress, Node test runner, or auto.
exclude Generated files, build outputs, vendored folders, snapshots, coverage, and framework output folders.

Task IDs should be stable and should follow the shape already consumed by project-management-board.

7. Rust Measurement Equivalents

7.1 Hotspots

The Rust lens ranks complexity hotspots using cognitive complexity, cyclomatic complexity, maintainability, effort, size, and density. The TypeScript React equivalent should rank risky source units across files, functions, components, hooks, reducers, route handlers, loaders, actions, and utilities.

Signals

  • Cognitive complexity.
  • Cyclomatic complexity.
  • Maximum nesting depth.
  • Conditional branch count.
  • Boolean expression complexity.
  • Async control-flow pressure.
  • Try/catch/finally density.
  • Function length and file length.
  • JSX density inside components.
  • Render branch complexity.
  • Mapped JSX regions.
  • Reducer case count.
  • Route module responsibility count.

Records

  • File records.
  • Function records.
  • Component records.
  • Hook records.
  • Reducer records.
  • Route module records.

Important Distinction

Component complexity should not be treated exactly like ordinary function complexity. A component with moderate function complexity can still be risky if its JSX tree has many conditional branches, nested render helpers, and mixed responsibilities.

7.2 Clones

The Rust lens detects token and AST-like clones. The TypeScript React equivalent should detect repeated utilities, components, hooks, form logic, reducers, tests, route scaffolding, and styling variants.

Clone Classes

  • Exact token clones.
  • Identifier-normalized clones.
  • Literal-normalized clones.
  • AST-normalized function clones.
  • AST-normalized component clones.
  • Repeated hook patterns.
  • Repeated test setup and assertions.
  • Repeated route modules or API handlers.
  • Repeated className or styling structures.

Scoring

  • Cross-file clones should score higher than same-file repetition.
  • Cross-feature clones should score higher than clones within a local family of variants.
  • Large clones should score higher than small clones.
  • Clones in tests should be reported separately because duplication tradeoffs differ in test code.
  • Generated code, snapshots, and compiled output should be excluded.

7.3 Escape Hatches

The Rust lens tracks unsafe blocks, FFI, raw memory, mutable globals, glob imports, and lint suppressions. TypeScript and React have a different kind of danger: places where code opts out of static safety, React rules, module boundaries, or browser safety.

Type-System Escapes

  • Explicit any.
  • Implicit any.
  • Unknown values used without narrowing.
  • Non-null assertions.
  • Type assertions.
  • Double assertions.
  • as any.
  • as never.
  • Overly broad object, Function, Record<string, unknown>, and Promise<any> surfaces.
  • Untyped JSON parsing.
  • Declaration shims that hide real dependency types.

Compiler and Lint Escapes

  • ts-ignore.
  • ts-expect-error.
  • ts-nocheck.
  • eslint-disable.
  • eslint-disable-next-line.
  • File-level rule suppressions.
  • Hook dependency suppressions.
  • Broad suppressions without a nearby explanation.

Runtime and Browser Escapes

  • dangerouslySetInnerHTML.
  • eval-like APIs.
  • Dynamic Function construction.
  • Direct DOM mutation inside React components.
  • Module-level mutable state.
  • Window globals.
  • Local storage used as hidden shared state.
  • Event buses and singleton registries.

7.4 Type Health

The Rust lens measures wide structs, large enums, broad method surfaces, and impl spread. The TypeScript React equivalent should measure structural risk in interfaces, type aliases, unions, classes, public exports, component props, hook return values, and generic APIs.

Signals

  • Wide interfaces.
  • Wide component prop types.
  • Large unions.
  • Large discriminated unions.
  • Deeply nested object types.
  • Many optional props.
  • Boolean prop combinations that imply hidden modes.
  • Recursive conditional types.
  • Deep mapped types.
  • Many type parameters.
  • Overloaded function signatures.
  • Exported functions with weak return types.
  • Types declared or augmented across many files.

React-Specific Type Health

  • Components with broad props and little composition.
  • Props mixing data fetching, rendering, interaction, styling, and routing concerns.
  • Callback props with unclear contracts.
  • Children typed too broadly where a structured slot would be clearer.
  • Hook return tuples with unclear positional semantics.
  • Context values with too many fields or too many update functions.

7.5 Code Locality

The Rust lens measures far dependencies, hidden coupling, public interface explicitness, test locality, churn, contributors, and layer violations. The TypeScript React equivalent should answer whether a developer can understand and change a feature locally.

Signals

  • Far imports across unrelated features.
  • UI importing infrastructure directly.
  • Feature modules importing private files from other features.
  • Deep relative imports.
  • Path aliases that obscure ownership boundaries.
  • Hidden coupling through stores, contexts, globals, singletons, local storage, and event emitters.
  • Provider values shared across too many modules.
  • Tests far from source.
  • No direct tests.
  • High churn.
  • Many contributors.
  • Frequent co-change with distant modules.

Test Locality Classes

  • Colocated test.
  • Nearby feature test.
  • External integration test only.
  • E2E evidence only.
  • Story evidence only.
  • No evidence.

7.6 Architecture Leverage

The Rust lens measures whether modules provide leverage or create ripple risk. The TypeScript React equivalent should identify stable primitives, brittle hubs, duplicated abstractions, and modules whose dependents keep working around them.

Positive Signals

  • Clear public API.
  • Well-typed exports.
  • Stable shared primitives.
  • Reusable design-system components.
  • Focused custom hooks.
  • Validated service boundaries.
  • Low co-change ripple despite broad use.
  • Self-contained leaf modules.

Risk Signals

  • High inbound reach with weak types.
  • Function-heavy surfaces with little domain modeling.
  • Many callers wrapping or adapting the same module.
  • Duplicated components near the abstraction.
  • High co-change ripple.
  • Shared modules importing feature-specific code.
  • Design-system components depending on application state.
  • Hooks that are shared widely but expose unstable contracts.

7.7 Correctness Catalog

The Rust lens discovers tests and optionally runs them. The TypeScript React equivalent should discover all meaningful correctness evidence and map it back to source modules, routes, components, hooks, and features.

Evidence Sources

  • Vitest tests.
  • Jest tests.
  • Node test runner tests.
  • Testing Library component tests.
  • Hook tests.
  • Integration tests.
  • Playwright tests.
  • Cypress tests.
  • Storybook stories.
  • Storybook interaction tests.
  • Framework route tests.

Test Status

  • Passed.
  • Failed.
  • Skipped.
  • Todo.
  • Unknown.
  • Not run.

7.8 Architecture Map

The Rust lens builds a graph with modules, dependencies, risk scores, correctness evidence, locality, leverage, artifact status, and category signals. The TypeScript React equivalent should produce the same kind of board-ready graph while reflecting frontend architecture.

Node Types

  • Package.
  • Feature.
  • Route.
  • Module.
  • Component.
  • Hook.
  • Context provider.
  • Store.
  • Service client.
  • Test group.

Edge Types

  • Static import.
  • Dynamic import.
  • Type-only import.
  • Route ownership.
  • Test reference.
  • Provider relationship.
  • Store usage.
  • Co-change relationship.

8. Additional Opportunities Beyond Rust Parity

8.1 React Health

  • Oversized components.
  • Components with too many props.
  • Boolean prop mode explosions.
  • Prop drilling depth.
  • Too many local state variables.
  • Too many refs.
  • Effects that derive state instead of synchronizing with external systems.
  • Effects mixing fetching, transformation, subscriptions, and state updates.
  • Custom hooks with mixed responsibilities.
  • Components defined inside components.
  • Nested render helpers.
  • Large conditional render blocks.
  • Repeated component variants.

8.2 Hooks and Effects Risk

  • Rules-of-hooks violations.
  • Exhaustive dependency violations.
  • Suppressions around hook dependency arrays.
  • Stale closure risk.
  • Missing effect cleanup.
  • Effects that should be event handlers.
  • Effects that should be render-time derivations.
  • Custom hooks returning unstable objects or callbacks.
  • Custom hooks hiding global coupling.

8.3 Render Performance Risk

  • Expensive computation during render.
  • Inline object, array, and function props passed to memoized children.
  • Large list rendering without virtualization hints.
  • Unstable keys.
  • Context providers with unstable values.
  • Context providers with broad values.
  • High-fanout components receiving frequently changing props.
  • Repeated derived state causing avoidable renders.
  • Overuse of memoization hiding unclear data flow.

8.4 Accessibility Health

  • Interactive elements without accessible names.
  • Click handlers on non-interactive elements.
  • Invalid ARIA usage.
  • Suspicious ARIA usage.
  • Missing form labels.
  • Missing or low-quality image alt text.
  • Dialog components without focus-management evidence.
  • Custom controls without keyboard interaction evidence.
  • Forms without visible or programmatic error association.

8.5 TypeScript Strictness Health

  • strict disabled.
  • noImplicitAny disabled.
  • strictNullChecks disabled.
  • noUncheckedIndexedAccess disabled.
  • exactOptionalPropertyTypes disabled.
  • noImplicitOverride disabled.
  • allowJs and checkJs posture.
  • skipLibCheck hiding dependency type issues.
  • Inconsistent tsconfig inheritance in monorepos.
  • Path aliases that make boundaries unclear.

8.6 Dependency Health

  • Import cycles.
  • Layer violations.
  • Deep imports into private internals.
  • Feature-to-feature private imports.
  • Duplicate dependency versions.
  • Heavy dependencies imported by client entry points.
  • Server-only modules imported into client paths.
  • Client-only modules imported into server paths.
  • Barrel files with excessive fan-in or fan-out.
  • Package boundary violations in workspaces.

8.7 Bundle and Runtime Opportunity

  • Large route-level dependency surfaces.
  • Missing dynamic import opportunities.
  • Unexpected client bundle dependencies.
  • Large shared chunks caused by convenience imports.
  • Duplicate utility libraries.
  • Polyfill pressure.
  • Design-system imports that pull in too much code.

9. Artifact Design

Each artifact should be useful alone and composable into the architecture map. Records should include stable IDs, file paths, line numbers where possible, scores, signals, evidence, provenance, and measurement confidence.

Artifact Contents
hotspots.json Ranked complexity and maintainability records for files, symbols, components, hooks, reducers, and routes.
clones.json Clone groups with engine, hash, classification, instances, score, and signals.
ts_escape_hatches.json Safety escape records grouped by module, file, kind, count, location, and severity.
type_health.json Type, interface, prop, generic, class, and public API structural health records.
locality_metrics.json Module-level dependency distance, hidden coupling, test locality, churn, contributor, and co-change records.
leverage_metrics.json Module-level reach, abstraction value, public surface, divergence, and ripple records.
react_health.json Component, hook, state, effect, context, and render-risk records.
dependency_health.json Cycles, layer violations, package boundaries, deep imports, barrels, and dependency pressure.
correctness_review.json Test summary, discovered tests, layer grouping, execution status, and correctness evidence.
test_catalog.json Flat list of discovered tests with IDs, names, paths, source mapping, status, and command hints.
map.json Dashboard-ready graph, category risks, evidence, summary metadata, and artifact status.

10. Risk Model

The first risk model should be simple and explainable. It should produce useful rankings without pretending to be mathematically perfect.

Risk Category Inputs
Maintainability Complexity, file size, component size, type complexity, clone pressure, weak boundaries.
Change Churn, contributor count, co-change ripple, defect history, test distance.
Correctness Failing tests, skipped tests, unknown tests, missing tests, hook violations, unsafe async patterns.
Render Performance Unstable render inputs, broad contexts, expensive render work, large lists, high fanout.
Accessibility Static accessibility findings and missing interaction evidence.
Architecture Cycles, layer violations, far dependencies, hubs, unclear public APIs, server/client boundary issues.

11. Measurement Confidence

Every major artifact should report confidence. This protects the dashboard from treating absent evidence as good news.

  • Whether source roots exist.
  • Whether tsconfig was found.
  • Whether the TypeScript project loaded successfully.
  • Whether type information was available.
  • Whether dependencies were installed.
  • Whether package manager detection succeeded.
  • Whether test runner detection succeeded.
  • Whether framework detection succeeded.
  • Whether git history was available.
  • Whether configured excludes were applied.
  • Whether unsupported file types were skipped.
  • Whether stale prerequisite artifacts were detected.

12. Framework Detection

The lens should work generically, then add richer interpretation when framework conventions are detected.

Framework Detection and Extra Interpretation
Vite Detect Vite config, src entry points, Vitest config, and client-only assumptions.
Next.js Detect app router, pages router, server components, client components, route handlers, layouts, and loading/error boundaries.
Remix Detect route modules, loaders, actions, nested routes, and route-level data boundaries.
React Router Detect route definitions, lazy modules, route loaders, and nested route ownership.
Expo / React Native Detect native platform files, app directory conventions, and React Native accessibility patterns.
Storybook Use stories as UI coverage evidence and map stories back to components.
Monorepo Detect npm, pnpm, yarn, Nx, Turborepo, and package workspaces.

13. Implementation Phases

Phase 1: Contract and Skeleton

  • Create the repository.
  • Define package metadata and CLI entry points.
  • Implement configuration loading.
  • Implement catalog output.
  • Implement artifact writing conventions.
  • Implement provenance and measurement confidence helpers.
  • Create fixture projects for tests.

Phase 2: Static TypeScript Measurements

  • Implement source discovery.
  • Implement TypeScript project loading.
  • Implement hotspots.
  • Implement escape hatches.
  • Implement type health.
  • Implement clone detection baseline.
  • Implement dependency graph baseline.

Phase 3: Correctness and Git Evidence

  • Discover tests across common runners.
  • Map tests to modules, components, hooks, routes, and features.
  • Optionally run tests.
  • Collect git churn, contributors, defect commits, and co-change data.
  • Feed correctness and git evidence into locality and map artifacts.

Phase 4: React-Specific Measurements

  • Implement component health.
  • Implement hook and effect risk.
  • Implement render performance risk.
  • Implement context and state coupling analysis.
  • Implement accessibility health baseline.

Phase 5: Architecture Map

  • Combine hotspots, correctness, locality, leverage, React health, and dependency health.
  • Produce board-compatible graph nodes and edges.
  • Group nodes by package, feature, route, or folder.
  • Expose category risk scores and evidence.
  • Represent unknown and stale artifacts honestly.

Phase 6: Dashboard Integration

  • Verify project-management-board can consume the catalog.
  • Verify each task can run individually.
  • Verify artifacts appear under the expected output directory.
  • Verify the architecture map renders with meaningful groups, nodes, edges, scores, and signals.

14. Recommended MVP

The first useful version should establish the contract and deliver trustworthy static insight before attempting deeper browser, bundle, and runtime analysis.

  1. Config loading.
  2. Catalog output.
  3. Hotspots.
  4. Escape hatches.
  5. Type health.
  6. Dependency health.
  7. Correctness catalog.
  8. Architecture map.

React health, render performance, accessibility, and bundle intelligence should follow once the base graph and artifact model are stable.

15. Validation Strategy

  • Fixture with simple TypeScript utilities.
  • Fixture with React components and hooks.
  • Fixture with deliberate type escape hatches.
  • Fixture with duplicate components and utilities.
  • Fixture with dependency cycles and layer violations.
  • Fixture with colocated and distant tests.
  • Fixture with Next.js-style server/client boundaries.
  • Snapshot tests for JSON artifact shape.
  • Golden tests for scores and signal labels.
  • Integration test for catalog plus measure all.
  • Dashboard smoke test through project-management-board.

16. Initial Tooling Bets

  • TypeScript compiler API: project loading, source files, symbols, types, exports, imports, and diagnostics.
  • typescript-eslint: ESTree traversal, typed lint-style analysis, and rule-compatible services.
  • eslint-plugin-react-hooks: hooks correctness and React Compiler diagnostics.
  • dependency-cruiser: dependency graph, cycles, and boundary validation.
  • jscpd: initial clone detection baseline.
  • Vitest, Jest, Playwright, and Cypress detection: correctness catalog and optional execution.

17. Open Decisions

  • Whether implementation should be pure Node/TypeScript or a Python CLI wrapping Node analyzers.
  • Whether to create a shared cross-lens artifact schema package now or later.
  • How much framework-specific logic belongs in core versus adapters.
  • How strict first-version risk thresholds should be.
  • Whether accessibility should be static-only at first or include optional browser checks later.
  • Whether bundle analysis belongs in the first release or a later performance lens.
  • Whether React Native should be first-class in the initial scope or treated as a later adapter.

18. Definition of Done

  • A TypeScript/React project can add a local lens config.
  • The lens can print a catalog with stable task IDs.
  • The lens can run all MVP measurements.
  • The lens writes board-compatible JSON artifacts.
  • The architecture map includes modules, edges, scores, signals, evidence, and artifact status.
  • The dashboard can run tasks and display results without hardcoding TypeScript-specific task metadata.
  • The fixture suite proves the core measurements are deterministic.
  • The README explains repository boundaries, quick start, artifacts, and dashboard integration.
